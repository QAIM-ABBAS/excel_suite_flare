# Cloudflare Pages + Workers Deployment Plan (No Credit Card Required)

## Overview

Deploy frontend to **Cloudflare Pages** and backend API to **Cloudflare Workers** with D1 database only (no R2 needed).

---

## Key Change: No R2 Storage

Instead of storing files in R2, we'll process files **in memory** and return results directly to the client as downloads. This eliminates the need for R2 entirely.

### How it works:
1. User uploads file(s) to Worker
2. Worker processes files in memory
3. Worker returns processed file as a download response
4. No server-side file storage needed

## Implementation Steps

### Step 1: Update Worker to Process In-Memory

The key change is that instead of uploading files to R2 and downloading them later, we'll:
1. Receive file uploads
2. Process them entirely in memory
3. Return the result as a download response

### Step 2: Update API Routes

Modify the Worker routes to:
- Accept file uploads via FormData
- Process files using in-memory Excel/CSV utilities
- Return processed files as binary responses

### Step 3: Update Frontend API Calls

The frontend will need to:
- Upload files to the Worker
- Receive processed files as downloads
- No more preview/history (since files aren't stored)

### Step 4: Deploy

1. Deploy Worker to Cloudflare
2. Deploy frontend to Cloudflare Pages
3. Configure CORS

## API Changes

### Before (with R2)
```
POST /api/tools/merge
- Upload files to R2
- Return { downloadUrl: "/api/tools/download?file=xxx" }

GET /api/tools/download?file=xxx
- Download from R2
```

### After (in-memory)
```
POST /api/tools/merge
- Process files in memory
- Return binary response with Content-Disposition header

No separate download endpoint needed
```

## Limitations

1. **No file history** - Since files aren't stored, history won't work
2. **No preview** - Can't preview files after processing
3. **File size limits** - Workers have 10MB request size limit
4. **Processing time** - Workers have 30 second CPU time limit

---

## Alternative: Use D1 for History Metadata Only

We can still track processing history in D1, just without the actual file content:
- Store filename, tool used, timestamp
- Store metadata only (not the file itself)

This gives users a history view without needing file storage.
